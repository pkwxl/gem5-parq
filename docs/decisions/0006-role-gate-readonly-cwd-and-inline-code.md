# 0006 — 角色门三处修补：只读误杀、粘性 cwd、解释器内联代码

- 状态：已采纳（2026-07-22）
- 相关：[0003 角色门三处裁定](./0003-role-gate-scons-merge-and-repo-config.md)、
  [0004 保留核](./0004-reserved-cores-single-source-and-idle-check.md)、
  `.claude/hooks/role-gate.py`、`CLAUDE.md`「Research role workflow」

## 1. 背景

三处都是在真实会话里被撞出来的，不是审查想出来的。前两处是**误判**（门拒绝了本该
放行的、或放行了本该拒绝的），第三处是门自己 docstring 里承认的「护栏不是沙箱」边界
——本记录把它从「完全不管」收窄到「看得见的部分要管」。

## 2. 缺陷 A —— 只读命令被判成「跑实验」

`ls -l build/X86/gem5.opt`、`du -sh .../gem5.opt`、`wc -l .../gem5.opt`、
`grep -rn scons docs/` 在主树全部被 `deny`，理由是「主树不跑 gem5 / 主树禁构建」。
纪律检查用的是 `search` 而非命令位锚定，所以**路径里出现二进制名**就足以触发。

更糟的是它**不一致**：`find /workspace/shm -name 'gem5.opt'` 反而放行——因为
`unquoted()` 会把引号内容抹掉。同一件事，加不加引号结论相反。那不是纪律，是抽签，
而且它训练出的行为是「被拒了就加个引号再试」，恰好是最坏的一种适应。

**决策**：加一份只读命令白名单（`READONLY_CMD`：`ls`/`stat`/`du`/`find`/`wc`/`cat`/
`grep`/`diff`/`readlink`/`git log|show|diff|status|…` 等），命令位命中即在**执行类**
纪律之前返回——`scons`/`gem5` 两块跳过。保留核（`CPU_LIST`）检查**不在**白名单之内：
`taskset -c 54 <任何东西>` 都是在占核，跟它读不读无关。

**否决的备选**：把所有纪律正则改成命令位锚定。更彻底，但要逐条重写并重验九条规则，
而真正出问题的只有「提到 vs 执行」这一类；白名单是最小且可测的修法。

## 3. 缺陷 B —— 相对路径按错误的树解析

`check_path()` 把相对路径拼在 `CLAUDE_PROJECT_DIR`（会话的项目根）上。但 **Bash 工具
的工作目录跨调用是粘的**：一次 `cd /workspace/gem5-wt/<branch>` 之后的每一条命令都在
那棵树里执行，而门仍然按主树解析。于是 `sed -i docs/specs/INDEX.md` 会被当成「PI 改
主树的 INDEX」放行，实际改的是**另一个 worktree 的 INDEX**——同时违反「INDEX 只在主树
维护」和「不得跨树写」两条红线。

2026-07-22 的 PI 会话里这件事真的发生了，只因为脚本里一条 `assert` 先失败才没写下去。
**门当时给的是 `allow`。**

**决策**：`main()` 从 payload 取 `cwd`，作为相对路径的解析基准传给
`check_path`/`gate_bash`；缺失或非绝对路径时回落到 root（与旧行为一致）。**不要求该
目录存在**——门在命令执行前运行，而「shell 自称在哪」正是要判的东西。

树与角色的判定**不变**，仍按 `CLAUDE_PROJECT_DIR` 与该树的 `.active-role`：会话的身份
不该因为 `cd` 而改变。改的只是「这条路径落在谁的地盘」。

## 4. 缺陷 C —— 解释器内联代码完全不受检

`python3 -c "open('docs/specs/INDEX.md','w')"` 与 `python3 - <<'EOF' … EOF`：路径不在
命令位上，`PATH_WRITERS`/`REDIRECTS` 提取不到，heredoc 正文还被 `segments()` 整段剥掉。
门此前对这类调用一律 `allow`。§3 那次事故用的正是 heredoc 形式。

**决策**：识别「解释器 + 内联代码」（`python|perl|ruby|node|sh|bash|awk` 后跟 `-c`/`-e`
或裸 `-`），命中后扫**原始命令串**（heredoc 未剥离）里长得像路径的 token，按 §3 的
`base` 解析后过 `check_path`；若某个路径本会被 `deny`，整条降级为 **`ask`** 并指出是
哪个路径。

为什么是 `ask` 不是 `deny`：门看不进代码，分不清那是读还是写。跨树**读**是正当的
（比对两份 spec），一律 `deny` 会挡掉正常工作；`ask` 把判断交给唯一知道意图的人。

**这不是沙箱，也不打算是**。`python3 -c "import os;os.system(...)"`、变量拼路径、
`base64` 解码后 `exec`——都能绕过。它挡的是**手滑**，与门的既定定位一致。

## 5. 后果与已知代价

- 白名单是列表法：漏列的只读命令仍会被误杀（表现为多一次确认，方向安全）；
  错列成只读的命令则会绕过执行类纪律——所以名单里**只放确定不写、不占核的命令**。
- `cwd` 依赖 harness 提供该字段。若某个版本不给，回落到旧行为——**静默地**回落。
  回归里有对应用例（不传 `cwd` 时行为不变），但没有任何东西会在字段消失时报警。
- 内联代码的路径扫描会有误报：一段只是**读**别处文件的代码也会被降级为 `ask`。
  可接受——频率低，且 `ask` 不阻断。
- 三处修补都不改写权矩阵本身，`use-role` 的 chmod 门不受影响。

## 6. 配套的散文纪律

机制只覆盖看得见的部分，剩下的靠纪律：**文件操作一律用绝对路径**（写进 `CLAUDE.md`
的角色工作流一节）。`cd` 的粘性是 §3/§4 两次事故的共同来源，而绝对路径让 §3 那类
误判在源头上不可能发生——这条比门的任何一处修补都更根本。

## 7. 何时应重新审视

- 若「加引号绕过」的行为再次出现（无论针对哪条纪律），说明白名单没覆盖到真实用法，
  应回到 §2 否决的那个备选：把纪律正则全部改成命令位锚定。
- 若内联代码的 `ask` 频繁到让人条件反射地确认，就该收紧触发条件（例如只在代码里出现
  写操作关键字时才降级），而不是取消它。
- 若 harness 停止提供 `cwd`，§3 的修补会静默失效——届时需要改成从别处获取，或把
  「相对路径一律 `ask`」作为兜底。
