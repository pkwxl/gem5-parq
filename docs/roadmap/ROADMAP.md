# 研究路线（PI 维护）

本文件是 PI 的产物，记录**当前阶段目标与优先级**。逐个研究的技术细节在
`docs/specs/S-NNN-*.md`，活跃状态在 `docs/specs/INDEX.md`——这里不复述，只写
「现在攻哪个、为什么、什么算成」。

## 长期目标

在百核以上的服务器上，对百核以上规模的系统模拟任务，取得相对现有单
`EventQueue` 串行 gem5 **超过 50x** 的真实 wall-clock 加速，同时保持时序精度
（或给出精确量化、经过深思熟虑的放宽）。

**与当前实测的差距（如实记录，2026-07-22）**：目前唯一一份在真实小时尺度、
均衡检查点上跑出的干净三臂数字是 S-017 的 **real speedup ≈ 0.459x**——并行臂
比 baseline **慢 2.18 倍**。距离 50x 有**两个数量级以上**，且方向是反的。此前
流传的 0.91x（S-009 §27）测自 `MAX_TICKS=2e8` 的窗口，S-013 §9 事后查明该窗口
只跑了 74588 条指令、大概率还在客户机线程启动阶段，不构成对当前工作点的有效
测量；以 S-017 为准。

这个差距不作为放弃目标的理由，但**任何阶段性汇报都必须以 0.459x 为基准数字**，
不得引用 0.91x 或 Internal speedup 代替 Real speedup。

## 当前阶段目标

**用最小的 gem5 改动，在 4 核模拟上取得较高加速比。** 这一阶段的意义是摸清
「最小改动」的边界在哪里、代价有多大，为后续放大到百核规模提供依据。

口径固定为 `CLAUDE.md`「三臂比较」：headline 数字是 **Real speedup =
Baseline / Current Parallel**，Internal speedup 不得代替它。

**阶段性成功判据**：在 S-017 同一条件（均衡检查点、`MAX_TICKS=1.34e12`、
tracing 关闭）下，Real speedup **跨过 1.0x**。跨不过去而原因被定量查清，
同样算这一阶段有结论——负面结果按 `CLAUDE.md` 工作风格如实落地。

## 当前的主要矛盾

S-017 §6 已经把 0.459x 定量归因完毕，这决定了下面的优先级排序：

- parallel 比 baseline 慢 915.67s；跨域屏障的到达时间差（`spread_ns`）总和
  **1097.34s**，是该差距的 **1.20 倍**——屏障忙等本身就足以、甚至超额解释全部
  减速，**不需要再假设锁或其他机制**（S-012 §19 也独立测得锁等待占比 <0.15%）。
- 其中约 **73%（~900s）** 是「固定税」：4.024 亿次跨域同步里 98.5% 每次都要付
  1–4 微秒。
- 其中约 **16%（~200s）** 来自 847 次极罕见但巨大的长尾停顿（最大单次 10.16 秒），
  是 S-012 §18 同类现象的第二次独立观测，两次都未查明根因。

结论：**当前瓶颈是每次跨域同步的固定成本，不是正确性、不是锁竞争。**
提速的唯一直接手段是摊薄它——即 OPEN-ISSUES A1 的「出路 3」。

**为什么 S-019 排在出路 3 之前**：S-018 §6.2 定位的下溢发生在共享的上游 stats
代码（`AvgStor`），并行臂的**任何时间加权统计都因此不可信**。出路 3 的全部工作
都要靠测量来判定成败，先修统计是为了不让后续实验数据带毒。它本身不产出加速比，
这一点在立项时就要写明，不要事后拿它充抵进展。

## 里程碑与决定记录

按日期倒序追加。不利结论同样记在这里——负面结果是研究记录的一部分。

### 2026-07-22 — 确立以 0.459x 为基准，出路 3 立为 P0（PI 决定）

S-017 交付了首个真实小时尺度的干净三臂数字（overhead ratio ≈ 1.002、real
speedup ≈ 0.459x、internal ≈ 0.460x），并在 §6 把减速定量归因到跨域屏障的固定
同步税。据此做出四项决定：

1. 项目基准数字改用 0.459x，0.91x 作废（窗口无效，S-013 §9）。
2. 出路 3（OPEN-ISSUES A1）从「缺数据不该立项」解封——S-012 §19 与 S-017 §6
   已经补齐了它缺的关键路径数据，升为 P0。
3. S-019（`AvgStor` 跨域下溢）先于出路 3 执行，避免后续测量带毒。
4. S-018 结题：O3 从 Atomic 检查点 restore 已跑通，`m_buf_msgs` 分歧已根因
   定位并移交 S-019；O3 本身能带来多少并发收益是新问题，另行立项。

### 2026-07-22 — 保留核策略成文，S-020 占号（PI 决定）

一次 Architect 会话把角色门的三处缺口和保留核的三处缺口裁定完毕（[决策 0003](
../decisions/0003-role-gate-scons-merge-and-repo-config.md)、[0004](
../decisions/0004-reserved-cores-single-source-and-idle-check.md)）。对研究路线有直接
影响的是后者：保留核的核号收进 `util/roles/reserved-cores` 单点定义；新增 `check-cores`
skill，在开跑前**实测**保留核是否空闲——因为容器内 `ps` 看不见宿主/其他容器绑在隔离核
上的任务，而这个盲区**第一次跑就复现了**（10 个保留核 100% 忙，本容器 `ps` 里干干净净）。

据此三项决定：

1. 占用状态按**实测**判定且**忙是粘性的**（测到忙即假定持续忙，直到下次测到空闲）；
   空闲核够用就开跑，默认不因「可能打扰同机其他实验」而阻塞。
2. **跨臂借核默认放行**——串行臂只有 2 个核、极易被占满，不许借会让大量实验白白排队。
3. 代价是跨臂即跨 NUMA node，而这个偏移量从未测过。**立 S-020 量化它**（P2）：它不产出
   加速比，是给第 2 条做量化背书；一旦实测出现无法解释的 node 0/1 差异就升 P1，因为那时
   所有借过核的历史数字都要重审。

### 2026-07-22 — 引入六角色研究工作流（PI 决定）

`CLAUDE.md`「Research role workflow」+ `docs/roles/**`。角色按工作树划分
（主树 pi/architect，worktree researcher/experimenter/implementor/debugger），
写权由 `util/roles/use-role` 的 chmod 门和 `.claude/hooks/role-gate.py` 双层
机制强制，不靠协议文字。动机是这个项目已经出现过多次「结论随窗口长度漂移」
「不利结果只在对话里提一句」的模式，需要机制而非自律来兜住。
