# S-020 — 跨臂借核的 NUMA 偏移量（串行臂 node 0 vs node 1）

> 状态：**初版（Architect，2026-07-22）**。占号见 `INDEX.md` S-020 行，优先级 P2
> （`docs/roadmap/ROADMAP.md`）。分支 `s020-numa-node-offset-serial-arm` 与 worktree
> 由 PI 在本初版落盘后创建。
>
> **本 spec 不产出加速比数字**，也**不适用** `CLAUDE.md` 的三臂制——理由见 §4.1。

## 1. 背景与目标

### 1.1 承接关系

[决策 0004](../decisions/0004-reserved-cores-single-source-and-idle-check.md) §3.1
把**跨臂借核设为默认放行**：串行臂的保留核只有 2 个、极易被占满，不许借会让大量实验
在一台大部分核空闲的机器上白白排队。该决策同时记下它的代价——两组保留核分属不同 NUMA
node（`SERIAL_ARM_CPUS` 在 node 0，`PARALLEL_ARM_CPUS` 在 node 1），**而借核带来的
`hostSeconds` 偏移方向与幅度从未测量过**。协议目前只要求「记录用了哪个 node」：这让
差异**可见**，但没有回答它**有多大**，因而也无法判断既有结论受不受影响。

本 spec 就是那个量化。

### 1.2 一次事实核查改变了风险面

写本初版时核对 S-017 原文，发现 0004 §3.1 初稿的一句话不成立，已在该决策里当场订正
（见其 §3.1「事实订正」条）。事实是：

| S-017 §2.2 的臂 | 绑核 | NUMA node |
|---|---|---|
| Baseline | `108` | **node 1** |
| Current Serial | `109` | **node 1** |
| Current Parallel | `100-107` | **node 1** |

**项目当前的基准数字（real speedup ≈ 0.459x）整套测自 node 1**——S-017 早就在做本决策
后来才合法化的跨臂借核。这把风险面挪了位置：

- **不是**「S-017 内部被污染」。三臂同在 node 1，任何 node 相关的偏移对三条臂同向作用，
  在 `Baseline / Current Parallel` 这个比值里大部分抵消。S-017 的内部一致性不受本问题
  威胁。
- **而是跨 spec 比较**。S-008/S-009 的串行臂用的是 `54`（node 0）。凡是把不同 spec 的
  `hostSeconds` 直接并列的地方，都可能在不知情的情况下跨了 node。

### 1.3 目标

1. 给出 **`R_numa` = node 1 的 `hostSeconds` / node 0 的 `hostSeconds`**（同一二进制、
   同一 checkpoint、同一工作点、串行臂）及其重复性。
2. 据此判定 0004 §3.1 的默认放行是否需要收紧（0004 §6 已预写升级触发器）。
3. 给跨 spec 比较提供一个明确依据：要么「偏移不显著，可直接比」，要么「偏移显著，
   跨 node 的数字不得当作同条件重复」。

## 2. 研究点拆解

编号供 Researcher 逐个深化；R1 是主目标，其余按序。

- **R1（主）—— 偏移的量级、方向、离散度。** 同一工作点下，串行臂在 node 0 保留核与
  node 1 保留核上的 `hostSeconds` 差多少？方向是哪边快？重复跑的组内离散度是否小于
  组间差异（否则测到的是噪声不是偏移）？
- **R2 —— 偏移是否随窗口长度漂移。** `R_numa` 是可以当常数用的机器属性，还是随
  `MAX_TICKS` 变化？**本项目已被「结论随窗口长度漂移」坑过三次**（S-012 §19 关键路径
  份额方向性反转、S-013 §9 窗口只跑到客户机启动阶段、S-014 崩溃发生在已验证窗口之外），
  所以这一条不是可选的稳健性检查，是本项目的必查项。
- **R3 —— 偏移对比值的影响。** headline 是 Real speedup = Baseline / Current Parallel。
  若两条臂同向受影响，比值可能远比绝对值稳健。**注意本 spec 只能部分回答 R3**，
  原因见 §3.4。
- **R4（初版新增）—— 历史 pin 核清点。** 逐个 spec 核对实际用过的 `taskset` 核号与其
  NUMA node，标出哪些结论是跨 node 比较出来的。这是**纯文档审计，不需要机时**，但它
  决定了 R1 的结果要回头修正哪些既有数字。加这一条是因为 §1.2 的核查已经说明：这类
  跨 node 比较**确实发生过**，且没有任何地方记录它。
  > PI 注：占号时 `INDEX.md` 只列了 R1–R3，R4 是初版补入的范围扩展，请 PI 决定是否
  > 同步 `INDEX.md` 的 S-020 行。

## 3. 设计思路

### 3.1 两条臂

除绑核外**一切参数相同**，这是本设计唯一的自变量：

| 臂 | 绑核 | NUMA node |
|---|---|---|
| **N0** | `SERIAL_ARM_CPUS` 里的一个空闲核（当前值下为 `54` 或 `55`） | node 0 |
| **N1** | `PARALLEL_ARM_CPUS` 里的一个空闲核，建议沿用 S-017 用过的 `108`/`109` | node 1 |

**共同参数**（全部沿用 S-017 §2.1/2.2，用户 2026-07-22 拍板「沿用 S-017 全套条件」）：

| 项 | 值 | 依据 |
|---|---|---|
| 二进制 | `main` 的 `build/X86_MESI_Three_Level/gem5.opt`，`PARALLEL_EVENTQ=0` | 即 S-017 的 Current Serial 臂。选它而非 Baseline 二进制：两者都能测 node 效应，而这一份在 worktree 里重建成本最低 |
| checkpoint | `/workspace/gem5-ckpt/x86-threads-balanced3-roi-classic` | S-017 §2.1 |
| `MAX_TICKS` | `1340000000000` | S-017 §2.1 |
| `EVENTQ_CRITPATH_TRACE` | unset（关） | S-017 §2.1：tracing-on 的 `hostSeconds` 与 tracing-off 不可比 |
| `STAT_DUMP_PERIOD` | `2000000000` | S-017 §2.1，进度可见性 + 被迫中止时的部分数据 |
| 驱动脚本 | `docs/refs/scripts/x86_fs_mesi3_parallel_eventq.py`，**原样使用** | 只改 `taskset --cpu-list`，不碰脚本 |

**本 spec 不需要任何代码改动**——没有 Implementor 任务。

### 3.2 编排：必须交替，不得分组

`N0 N1 N0 N1 …` 交替跑，**不要** `N0×3` 再 `N1×3`。理由：这台机器上有别的容器/宿主
任务来去（0004 §1「一次现场验证」记录了 10 个保留核被外部占满而本容器 `ps` 全干净），
分组编排会把「机器状态随时间的漂移」整段算进后跑的那一臂，伪装成 node 效应。交替编排
让漂移在两臂间摊平。

### 3.3 每次跑之前必做

按 0004 §3.1：跑 `check-cores`（`python3 .claude/skills/check-cores/sample-cores.py`），
把逐核忙碌率、实际选用的核号与其 NUMA node、以及**同机当时有没有别的实验在跑**记进
结果表。最后一项是本次数字的已知扰动源，不是可选项——尤其因为本 spec 测的偏移量级
（预期个位数百分比）可能与外部负载造成的扰动同量级。

### 3.4 为什么并行臂不可测，以及 R3 因此只能部分回答

并行臂需要 8 个核，而 node 0 的保留核只有 2 个——**在保留核范围内把并行臂搬到 node 0
不可行**；借非隔离核（`BUILD_CPUS`）会引入比待测量本身更大的噪声。这一条在占号时即已
确认，写在这里是为了让后来的人不必重新发现。

因此 R3 只能这样回答：S-017 已经有「node 1 serial / node 1 parallel」这个纯 node-1 比值；
本 spec 的 N0 臂提供「node 0 serial」。两者相除得到的是一个**混合比值**，它与 S-017 纯
node-1 比值的差，等价于把 `R_numa` 代入比值分子后的效果。**这不是对「并行臂本身的 node
敏感度」的测量**，Researcher 写计划时必须把这个局限原样写进去，不得把混合比值当成完整
的 R3 答案。

### 3.5 排期上的硬约束

R1 的 N0 臂**必须**用 node 0 的串行保留核——这是本项目里**唯一一个不能靠跨臂借核绕开
的实验**。占号时 `54-55` 正被另一次三臂测量 100% 占满，按 0004 §3.1 的粘性规则，要靠
重复跑 `check-cores` 才能确认释放。

若长期不释放，Researcher 可在计划里写一个**明确降级的**探索方案：用 `BUILD_CPUS` 里
node 0 的空闲核跑一次 N0，结果**必须标注「非隔离核，仅供量级参考，不构成 R1 的答案」**，
且不得用它触发或排除 0004 §6 的升级。

## 4. 验收标准（预先写死，不得事后修改）

### 4.1 口径

**本 spec 不适用三臂制。** `CLAUDE.md`「三臂比较」回答的是「这个 fork 的并行化有没有
带来真实加速」，本 spec 回答的是「测量平台本身的一个系统性偏移有多大」，两者的臂定义
不同、比值含义也不同。硬套三臂模板会得出无意义的数字。

**headline** = `R_numa = median(hostSeconds | N1) / median(hostSeconds | N0)`，
连同两组的**离散度**（极差与样本标准差都报；样本量小，不做显著性检验的花活）一起给出。
单独报 `R_numa` 而不报离散度，视为未完成。

### 4.2 判定（用户 2026-07-22 拍板 5% 阈值）

| 条件 | 判定 | 后续动作 |
|---|---|---|
| `\|R_numa − 1\| > 5%` **且**组间差异大于两组离散度 | **偏移显著** | 触发 0004 §6 的重审：S-020 升 P1；0004 §3.1 的默认放行需收紧（跨臂借核须同时给同 node 对照，或降级为需计划显式批准）；R4 清出的跨 node 历史比较逐条加限定说明 |
| `\|R_numa − 1\| ≤ 5%` 且离散度不掩盖它 | **偏移不显著** | 0004 §3.1 维持不变；把该结论写进 0004 §5 的代价条款，跨臂借核从此有量化背书 |
| 组内离散度 ≥ 组间差异（噪声主导） | **无定论** | 不得二选一地报成上面任一条。增加重复次数或换更安静的时段重跑；若仍无定论，如实记为负面结果 |

选 5% 的理由（预先写明，防止事后调阈值）：它远小于项目关心的量级（当前 real speedup
0.459x，与 1.0x 差一倍以上），又大到不会被主机噪声淹没。

### 4.3 无论结果如何都必须记录

1. 每次跑实际用的核号**及其 NUMA node**；
2. 该次跑之前 `check-cores` 的逐核实测值；
3. 同机当时是否有别的实验在跑；
4. 每臂是否**干净跑到** `MAX_TICKS` 边界（退出码、日志无 assert/panic/段错误）——
   被 `timeout` 杀掉的跑其 `hostSeconds` 只能作参考，不进 median；
5. 不利结果当场写进本文件，同一会话内完成（`CLAUDE.md` 工作风格第一条）。
