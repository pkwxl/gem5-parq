# S-008 — FS 串行 vs 并行实测：当前工作点的真实加速比

## 15. 定窗 A/B：串行首次补齐，Q=300 下并行反而慢 ~3x

S-007 §12.6 测了 FS **cv vs spin**（1.41x），但项目目标定义在**并行 vs 串行**
上，而 FS 侧串行臂在那次实验里从未跑过。本节补上这一臂。

### 15.1 方法

沿用 12.6 的定窗方法，同一二进制（`build/X86_MESI_Three_Level/gem5.opt`，
已含自旋屏障 + pythonDump 修复）、同一检查点
（`CHECKPOINT_DIR=/workspace/gem5-ckpt/x86-threads3-roi-classic`）、同一窗口：

- `MESI_Three_Level`、4 核、`MAX_TICKS=2e8`、`LINK_LATENCY=20`（脚本硬编码）、
  `SIM_QUANTUM_TICKS=300`（脚本硬编码）——6.67e5 个 quantum。
- 驱动脚本：`docs/refs/scripts/x86_fs_mesi3_parallel_eventq.py`。
- **串行臂**：`PARALLEL_EVENTQ=0`，`taskset -c 54` 整进程绑核。
- **并行臂**：`PARALLEL_EVENTQ=1 HOST_PIN_CPUS=92,93,94,95,96,97,98,99
  EVENTQ_BARRIER_MODE=spin`（8 个 EventQueue：核 1-4、L3=域 5、两个内存控制器
  =域 6-7）。
- 与 12.6 的一个重要差异：这次两臂都跑在内核 `isolcpus=54-55,92-111` 隔离出的
  核上（串行用 node0 的 54-55，并行用 node1 的 92-111），12.6 当时绑的是
  **未隔离**的 node0 8-15（会被 OS 调度器抢占/加噪声）。两次实验的绝对
  host-second 数字因此**不可跨会话直接比较**——见 15.3 的单独说明。
- 2 轮（round1/round2），第一轮跑之前内核镜像/磁盘镜像资源还没缓存到
  `~/.cache/gem5`（首次下载+解压花了约 4 分钟一次性开销，之后的轮次都是秒级）。

| **spin**（新测，本节，隔离核） | 4.05 | 3.98 | **4.02 s** | 74062 |
| cv（12.6，未隔离核，仅作参照） | — | — | 57.1 s | 74062 |

| hybrid@200（12.6，未隔离核，仅作参照） | — | — | 40.4 s | 74062 |

**正确性读数**：四次新跑（serial×2、spin×2）`simInsts` 全部等于 12.6 的
74062，逐一致。更强的一条：`diff` serial-round2 与 spin-round2 的完整
`stats.txt`（排除 `hostSeconds`/`hostMemory`/`hostTickRate` 等宿主计时字段）
**逐字节相同**——Q=300 下的量子放宽在这个窗口内没有产生任何可观测的时序
或指令数偏差，串行/并行/cv/spin/hybrid 五种配置到此为止全部时序中性。

### 15.3 结果如实报告：这不是加速，是减速

**这是一个违背项目目标的测量，按工作方式要求原样记录**：在当前工作点
（Q=300、ll=20、这个 2e8-tick 窗口），并行 EventQueue **比串行慢**：

```
speedup = W_serial / W_spin = 1.32 / 4.02 ≈ 0.33x
```

也就是并行版本比串行慢了约 **3 倍**，不是 12.6 单独看 cv→spin 对比时给人
的"1.41x 更快"的印象——12.6 从未把 spin 和串行放在一起比过，1.41x 只是
"两种并行屏障互相比"，两者都仍然远慢于串行。

**为什么新测的 spin（4.02s）比 12.6 的 spin（40.5s）快了一个数量级**：最可能
的解释是 CPU 隔离——12.6 绑的 8-15 核没有 `isolcpus`，宿主 OS 调度器的抢占/
迁移噪声直接计入了 spin-wait 的墙钟时间；这次的 92-111 是内核隔离核，spin
路径不再跟别的进程/中断竞争。这本身可能是一个值得记住的独立杠杆（"隔离核"
本身就能把 spin 屏障开销砍掉一个数量级），但**不影响 15.3 的结论**——因为
这次的串行臂**同样**跑在隔离核上（54-55），15.2 的 serial/spin 对比是同条件
下测的，3x 减速是隔离核条件下的真实结果，不是噪声artifact。

### 15.4 缺口算术：抬高 Q 能不能补齐

沿用 S-004 §9.3 出路 1 的框架，现在有了真实测的 W：

- 当前 Q=300：6.67e5 个 quantum；
- classic iobus（IOXBar）跨域边固定后，Ruby 链路合法上限
  Q ≈ ll×333 = 20×333 = 6660（S-004 §9.3 path 1 的目标值）；quantum 数降到
  2e8/6660 ≈ 3.0e4，降幅 **22.2x**（吻合 §9.3 原先估的"22x quantum gap"）。
- 本节自测的每-quantum 屏障开销：`(4.02 − 1.32) / 6.67e5 ≈ 4.05 µs/quantum`
  ——与 S-004 §9.2/S-007 §12.6 在完全不同硬件上独立测出的 ~3µs/quantum 量级
  一致，互相印证。
- 把这个每-quantum 开销按 22.2x 降幅投影到 Q=6660（假设非屏障的固定开销——
  线程创建/加入等——不随 Q 变化）：
  - 屏障开销部分：3.0e4 × 4.05µs ≈ 0.12 s（从 2.70 s 降到 0.12 s）；
  - 非屏障固定开销部分 ≈ 4.02 − 2.70 = 1.32 s（不变，凑巧约等于串行耗时）；
  - 投影并行总耗时 ≈ 1.32 + 0.12 ≈ **1.44 s**；
  - 投影加速比 ≈ 1.32 / 1.44 ≈ **0.92x**——仍然 <1x，只是从明显减速
    （0.33x）抬到接近打平（0.92x）。

**结论**：单靠出路 1（抬高 Q）投影只能把这个工作点从 0.33x 抬到 ~0.92x——
接近打平，够不到 1x，更够不到项目目标里隐含的 3x。这和 SE 侧已有的结论
（S-004 §9.9、S-007 §13：加速区把并行从 0.59x 抬到 0.90x，同样"接近平价、
没有超过"）**是同一个物理图景的再次印证**：屏障成本摊薄后，下一个瓶颈是
每 quantum 的有用工作量太少。要真正拿到 >1x 甚至 3x，出路 1 和出路 3
（S-004 §9.3——每域塞更多工作，摊薄固定同步成本，比如整合更多核/L3
bank 到一个域）大概率要**一起做**，不能只做出路 1。这是投影，不是实测——
真正验证需要 S-009 把 Q 上限实现出来后重新跑这个 A/B。
